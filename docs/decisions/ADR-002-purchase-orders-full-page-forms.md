# ADR-002: فواتير المشتريات — نماذج صفحات كاملة بدل النافذة المنبثقة

**الحالة:** مقبول  
**التاريخ:** 2026-05-12

## السياق

كانت قائمة «فواتير المشتريات» (`PurchaseOrderList`) تدعم إنشاء وتعديل المستندات داخل **مودال** كبير يشبه نافذة الفواتير. ذلك يفرض:

- تمرير عمودين داخل ارتفاع شاشة محدود على الجوال.
- ازدواجية منطق الحفظ بين القائمة والنماذج المستقلة المستخدمة لفواتير العملاء (`InvoiceForm` على `/invoices/create`).

## القرار

1. نقل **إنشاء** و**تعديل** فواتير المشتريات إلى صفحات كاملة:
   - `GET /purchase-orders/create` → `PurchaseOrderForm` بدون معرّف.
   - `GET /purchase-orders/{purchaseOrder}/edit` → `PurchaseOrderForm` مع النموذج الممرَّر.
2. الإبقاء على `PurchaseOrderList` لـ: التصفية، العرض السريع، الحذف، والروابط إلى الصفحات والتفاصيل.
3. محاذاة **تخطيط البطاقات** مع نموذج الفاتورة: معلومات كاملة العرض → بنود كاملة العرض → ملاحظات موسّعة → إجمالي → حفظ.

## البدائل التي رُفضت أو أُجِّلت

| البديل | السبب |
|--------|--------|
| مودال فقط | لا يتماشى مع مسار الفواتير؛ صعوبة طباعة/مشاركة الرابط؛ ضيق على الشاشات الصغيرة. |
| Inertia/Vue منفصل | خارج المكدس الافتراضي المعتمد دون ADR إضافي. |

## العواقب

- **إيجابية:** مسارات واضحة؛ إمكانية إضافة اختبارات Feature على الصفحة؛ تجربة موحّدة مع فواتير العملاء.
- **سلبية:** ملفان إضافيان للعروض (`create`/`edit`) ومكوّن `PurchaseOrderForm` يجب صيانتهما مع تغييرات المجال.
- **الترتيب في المسارات:** يجب تعريف `create` و`edit` قبل `{purchaseOrder}` لتجنّب التعارض مع المعرف الحرفي.
