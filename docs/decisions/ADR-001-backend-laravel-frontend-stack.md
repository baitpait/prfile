# ADR-001: اعتماد Laravel للباكند واقتراح الفرونت

## الحالة
مقبول — **الباكند:** Laravel (PHP). **الفرونت:** التوصية الافتراضية أدناه (قابلة للتعديل بـ ADR لاحق إن انحرف التنفيذ بشكل كبير).

## السياق
- المنتج: تقارير، لوحات، وثائق عربية RTL، مستندات مالية، صلاحيات.
- الفريق اختار **Laravel** لخبرة الصيانة والمجتمع وEloquent والهجرات ونظام Auth/Policy الجاهز.

## القرار

### الباكند
- **Laravel** (يفضّل **PHP 8.2+**) مع:
  - **Form Request** للتحقق من المدخلات.
  - **Policies / Gates** للصلاحيات على التقارير والكيانات.
  - **Migrations** قابلة للعكس حيث أمكن؛ تعليق عند الحذف التدميري.
  - **Eloquent** مع SoftDeletes للبيانات الحرجة عندما ينطبق المخطط.
  - **Queues** للتصدير الثقيل (PDF/Excel) عند الحاجة.

### الفرونت (التوصية الافتراضية)
- **Laravel Blade** كهيكل صفحات + **Livewire 3** للتفاعلية + **Alpine.js** للسلوكيات الخفيفة + **Tailwind CSS** للتصميم و**RTL** (`tailwindcss-rtl` أو `logical` properties).
- **السبب:** Monolith قوي، سرعة في شاشات الإدارة والتقارير، نفس لغة التحقق والسياسات على الخادم، سهولة `dir="rtl"` وخطوط عربية من `docs/05_VISUAL_IDENTITY_AR.md`.

### البديل المعتمد هندسياً
- **Inertia.js + Vue 3** (مع **TypeScript** اختياري) إذا احتجتم SPA أغنى مع بقاء التوجيه والمصادقة من Laravel.
- **السبب:** تجربة مستخدم أقرب للتطبيقات الأحادية الصفحة مع الحفاظ على Laravel كـ API/Resource.

### غير موصى به كبداية
- فرونت منفصل **React/Next** بالكامل مع API فقط — مكلف للفريق الصغير ما لم يكن هناك سبب ADR (تطبيق عميل خارجي، فريق فرونت منفصل بالكامل).

## العواقب
- إيجابية: سرعة التطوير، تقارير Blade/Livewire، تكامل ممتاز مع Laravel Excel / DomPDF / Snappy.
- سلبية: Livewire يتطلب ضبطاً جيداً للأداء على الجداول الضخمة (pagination، lazy loading).

## البدائل التي رُفضت كبداية
- FastAPI/Nest فقط: رغبة صريحة بالـ Laravel.
- Flutter للويب: خارج النطاق الحالي.
